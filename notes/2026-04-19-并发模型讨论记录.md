# 并发模型讨论记录

## 背景

在讨论 Codex 自我进化插件时，发现一个会影响系统成败的关键问题：

- Hermes 更接近单主实例语境
- Codex 在真实使用中经常会在同一台机器上同时开多个实例
- 多个 Codex 可能在不同目录、不同项目、不同 branch 中并行工作

这意味着如果直接照搬 Hermes 的“全局 memory + 全局 recall + 全局 skill 自动沉淀”，会出现严重并发问题。

---

## 已识别的风险

### 1. memory 并发写冲突

如果多个 Codex 实例同时 background review：

- A 在 repo1 写 `MEMORY.md`
- B 在 repo2 写 `MEMORY.md`
- C 又在写 `USER.md`

可能出现：

- 覆盖
- 交错写坏
- 去重失效
- 局部事实被错误提升成全局事实

### 2. recall 污染

如果 recall index 全局共用：

- 多个实例会同时写入不同项目经验
- 后续 recall 会混入不相关项目的经验
- 排序和 stale 判断都会变差

### 3. managed skill 互相踩写

如果多个实例同时判断：

- 某个 skill 应该 patch
- 某个 skill 应该 active / retire

则会出现：

- patch 丢失
- 生命周期乱跳
- 同一 skill 被不同上下文同时改坏

---

## 讨论过的一个方向：协调者（coordinator）

讨论中曾提出：

- 多个 Codex 实例只提交 suggestion / event
- 单机有一个协调者统一串行写 memory / recall / skill

这个方向的优点：

- 更容易处理并发
- 更容易统一做幂等、锁、ownership 检查、状态转换
- 更容易限制最终写入路径

但用户当前明确表示：

> **不想要协调者。**

原因：

- 会引入额外复杂度
- 当前阶段不希望系统过早长成一个单机总控服务

因此：

- 本次讨论**不采用 coordinator 作为当前默认路线**
- 也**不把 coordinator 写入 design 作为当前方案**

---

## 当前更被接受的方向：按目录 / 项目拆

本次讨论中的另一个关键判断是：

> **按目录 / 项目拆，会明显减轻并发问题。**

虽然不能完全消灭并发，但可以把问题从：

- 多个 Codex 争同一套全局状态

降低成：

- 多个 Codex 只在同一项目内争同一套局部状态

这会显著降低风险。

---

## 按项目拆带来的好处

### 1. memory 冲突范围变小

如果改成项目级：

- project-a / `MEMORY.md`
- project-b / `MEMORY.md`

则：

- repo 级事实不再去争同一份全局 `MEMORY.md`
- 局部约定不容易污染其他项目

### 2. recall 污染变小

如果 recall index 主要是项目级：

- 大部分 recall 只在自己项目池里找
- 跨项目误召回风险会明显降低

### 3. managed skill 边界更清楚

如果 skill 主要按项目隔离：

- project-a 的 managed skills
- project-b 的 managed skills

则：

- 自动 patch 范围更可控
- 生命周期更容易稳定
- 不容易跨项目互相污染

---

## 一个可能的双层结构（仅作为讨论记录，不进入当前 design）

这次讨论里，出现了一个更合理的结构想法：

### 全局层

只保留真正全局的东西，例如：

- `USER.md`

### 项目层

每个 repo / cwd 自己维护：

- `MEMORY.md`
- recall index
- managed skills
- review suggestion / queue

这个结构的优点是：

- 全局层只承载慢变量
- 高频变化的自我进化资产落在项目层
- 比“所有东西全局共享”更符合 Codex 的实际使用方式

但注意：

- 这只是**本次讨论中浮现出的方向**
- 用户当前只是确认“不想要协调者”
- **尚未正式拍板把架构改成全局层 + 项目层**
- 因此本条先作为讨论记录保存，不直接回写 design

---

## 本次讨论当前结论

1. 并发问题是成败级问题，不是实现细节。
2. coordinator 虽然有助于并发治理，但当前用户不希望引入这层复杂度。
3. 按目录 / 项目拆，会明显减轻并发问题。
4. “全局用户层 + 项目自进化层”的双层结构是一个值得后续认真考虑的方向。
5. 当前这一轮先只记录，不直接改写正式 design。

---

## 后续再讨论的真正问题

如果后面要继续推进并发模型，真正需要拍板的是：

1. 第一版是否从“纯全局”切成“全局 + 项目级”双层结构
2. `USER.md` 是否继续保持全局
3. `MEMORY.md` 是否改成项目级
4. recall index 是否项目级优先
5. managed skills 是否默认项目级
6. 在没有 coordinator 的前提下，如何做文件锁、SQLite 并发写和幂等处理
